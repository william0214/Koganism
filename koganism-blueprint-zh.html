協議 1 · T 細胞反應
交叉驗證
當 Gemma 4 讀取新內容時，會與你現有的知識庫進行交叉比對。 如果新資訊與已確立的事實衝突，將觸發炎症反應—— 通知使用者進行人工審查。內容被隔離直到問題解決。
新資訊 → 與知識庫比對 → 有衝突？
是 → 🔴 隔離 + 通知使用者
否 → ✅ 進入消化階段
協議 2 · 抗原中和
來源強制驗證
在階段 3 中提取的每個論點都必須附帶來源連結。 沒有來源 = 潛在病毒。沒有來源的論點會被標記為 [未驗證] 並存放在 /quarantine/—— 在人工驗證前永遠不會合併到主知識庫。
規則：每個事實性論點 → [來源: URL]
缺少來源 → [未驗證 — 隔離]
隔離區在階段 5 每月審查。
最小可行實踐
01
單一信號源
從一個開始。HN 首頁，或五個你真正在乎的 GitHub repo。不是十個。一個。
聞是廉價的，吃是昂貴的。
02
猶豫 = 拒絕
如果你不確定某個東西是否達標，那它就沒有。猶豫本身就是信號未通過。
猶豫本身就是不達標。
03
一句話消化
讀完之後，你能說「這改變了我對 X 的看法」嗎？如果不能，消化失敗。
沒有那句話 = 沒有消化。
04
每月排泄
每月一天：刪除的比新增的多。這是最難養成的習慣，也是最重要的。
只增不刪的知識庫是負債。
05
代謝檢查
每週：你這個月攝取的東西有出現在任何專案或決策中嗎？零 = 零代謝。
不行動的知識不算數。
06
建造即消化
Karpathy 的真正方法：從零實作。輸出迫使輸入變得真實。不問學到什麼——問建了什麼。
不問學到什麼，問建了什麼。
